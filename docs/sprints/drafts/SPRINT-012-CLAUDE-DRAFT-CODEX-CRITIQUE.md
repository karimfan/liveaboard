# Codex Critique of `SPRINT-012-CLAUDE-DRAFT.md`

## Findings

1. **The `num_guests` update policy is internally inconsistent and likely wrong for spreadsheet re-imports** ([docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:106](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:106), [:333](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:333), [:344](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:344)).
Claude’s draft says spreadsheet import sets `num_guests` only on insert and preserves the existing value on update via `COALESCE(EXCLUDED.num_guests, trips.num_guests)`. That means a corrected spreadsheet cannot update expected guest counts for an existing trip, which undermines the “re-upload an updated spreadsheet” use case. The preservation rule makes sense for liveaboard.com imports, but not for spreadsheet imports, which are the authoritative source for this field in this sprint. The plan needs per-source behavior: preserve `num_guests` on liveaboard re-scrapes, but allow spreadsheet re-imports to update it when the column is present.

2. **Preview/commit state is underspecified and contradictory, which makes the spreadsheet flow unsafe to implement as written** ([docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:131](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:131), [:136](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:136), [:403](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:403), [:493](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:493)).
The draft describes preview returning JSON directly, then commit taking `{file_id, vessel_mapping}`, and later redefines commit as `{vessel_mapping, rows_to_skip, source_fingerprint}`. Those are materially different designs. If the server does not persist a preview artifact, the commit endpoint must trust client-sent rows and warning state, which is fragile and makes “diff before commit” semantics hard to defend. If the server does persist preview state, the plan should name the table/object explicitly and make `preview_id` the commit contract. Right now the document leaves the hardest part of the spreadsheet flow unresolved.

3. **The async `import_jobs` model is probably over-scoped for Sprint 012 and pulls in shutdown/runtime complexity that the intent did not require** ([docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:14](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:14), [:76](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:76), [:188](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:188), [:377](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:377)).
The draft adds a job table, goroutine runner, polling UI, recent-jobs list, graceful shutdown semantics, and failure recovery machinery just to avoid a 20–30 second blocking request for a single-boat scrape. That is a lot of infrastructure for a sprint that already has a second import path, preview UX, vessel mapping, and a schema change. The draft is coherent if we explicitly want background work as a platform investment, but absent that decision it looks like avoidable scope expansion. A synchronous liveaboard import with a clear “this may take ~30 seconds” state is much cheaper and keeps the sprint focused on the actual operator workflow.

4. **CSV/XLSX plus ambiguous-date override is a second scope expansion inside the same sprint** ([docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:24](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:24), [:217](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:217), [:255](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:255), [:348](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:348)).
The draft does not just add `.xlsx`; it also introduces locale-ambiguous date parsing and a UI override path where the operator chooses between two interpretations for rows like `05/06/2026`. That is a meaningful product and implementation branch, not a parser detail. Combined with background jobs, this makes the sprint much larger than it first appears. The document should either narrow to CSV-first with explicit accepted date formats, or treat XLSX and ambiguous-date override as follow-up work.

5. **The spreadsheet source-provider policy is presented as settled, but the chosen semantics deserve more scrutiny** ([docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:286](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:286), [:298](docs/sprints/drafts/SPRINT-012-CLAUDE-DRAFT.md:298)).
Using `source_provider = "spreadsheet"` as one per-org bucket is simple, but it bakes in a strong product rule: every future spreadsheet upload for a given boat becomes the same authoritative source and stale-deletes prior future rows from that bucket. That may be correct, but the draft treats it as obviously right without discussing whether operators might upload distinct schedule files intentionally. This is less a bug than an important policy decision that should stay visibly open until merge.

## Open Questions

1. If spreadsheet imports are authoritative for `num_guests`, should the store layer expose separate reconciliation paths for liveaboard vs spreadsheet instead of overloading one `TripScrape` upsert rule?
2. If the team keeps async jobs, is the recent-jobs UI actually valuable in Sprint 012, or can the scope stop at kickoff + single-job polling?
3. If XLSX remains in scope, do we really want locale-disambiguation UI now, or should the accepted input contract simply require ISO dates?

## Summary

Claude’s draft has the right product direction: native import hub, preview-first spreadsheet flow, explicit vessel mapping, and reuse of the existing scrape/store seams. The main issues are scope and semantics. I would tighten it by removing the job system unless the planner explicitly wants it, making preview persistence first-class, and splitting `num_guests` policy by source so spreadsheet re-imports can actually correct guest counts.
