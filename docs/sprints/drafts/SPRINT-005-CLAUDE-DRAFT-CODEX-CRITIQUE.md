# Codex Critique of `SPRINT-005-CLAUDE-DRAFT.md`

## Findings

1. **The proposed signup flow does not actually cover US-1.1 because it never explains where the organization name comes from or who creates the first org row.** The draft says signup becomes Clerk `<SignUp>` plus a webhook-created `users` row and `organizations` row ([SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:79), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:212), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:240)). That skips the current required input `organization_name` from Sprint 003 and assumes Clerk user signup and app-tenant creation are the same event. They are not. The sprint needs an explicit org-bootstrap flow: either our backend creates the local org first and then provisions the provider membership, or the signup UI collects org name and calls an app endpoint that creates both the local org and the provider-side org/membership in a controlled order.

2. **The draft regresses the current auth contract from same-origin cookie auth to SPA-managed bearer tokens without treating that as a major migration cost.** The architecture and frontend plan move `/api/*` auth to `Authorization: Bearer <Clerk JWT>` and remove `/api/logout` in favor of client-side `signOut()` ([SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:109), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:217), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:243), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:255)). The intent explicitly calls out that the frontend currently assumes same-origin cookies and `/api/me`, and that any provider should preserve that contract or migrate it deliberately. As written, this is a materially larger frontend/runtime rewrite, increases token-handling complexity in the SPA, and weakens the “minimal app churn” goal. A stronger plan would preserve server-validated cookie auth at the app boundary and keep `/api/logout` as a backend endpoint even if Clerk is the session issuer.

3. **Webhook-only user/org synchronization introduces a race where a valid newly authenticated user can hit `/api/me` before the local row exists.** The draft repeatedly relies on webhooks to create local users and organizations after signup, invite acceptance, and profile changes ([SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:79), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:87), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:147), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:287), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:327)). That is eventually consistent, but the request path is treated as if it were immediately consistent. A first login or invitation acceptance can easily beat webhook delivery and produce a 401 for a perfectly valid provider session. The sprint should require synchronous upsert-on-auth/callback, with webhooks and reconciliation as secondary repair paths.

4. **The provider recommendation leans on a cost argument that is factually shaky and omits Clerk’s B2B limits.** The draft says Clerk has a “10K MAU free tier” and scores it as `$0 (free ≤10K MAU)` ([SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:22), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:44), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:395)). As of May 1, 2026, Clerk’s pricing page advertises 50,000 included MRUs per app on the main plans, and the B2B add-on’s free tier includes 100 organizations with up to 20 members per organization before the enhanced add-on. That does not automatically disqualify Clerk, but it does mean the draft’s cost comparison is not reliable enough to anchor the provider choice. The recommendation should either be re-scored with current pricing and limits or de-emphasize price as the deciding factor.

5. **The migration drops too much local auth state too early, which makes validation and rollback harder than necessary even in dev-only cutover.** The schema plan drops `password_hash`, `email_verified_at`, `sessions`, and `email_verifications` in the first provider migration ([SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:153), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:298), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:375), [SPRINT-005-CLAUDE-DRAFT.md](/Users/karim/src/liveaboard/docs/sprints/drafts/SPRINT-005-CLAUDE-DRAFT.md:400)). Since this repo is still pre-customer, destructive cutover is acceptable, but there is still no reason to make the first migration maximally destructive. The safer shape is: add provider linkage and cut the runtime over first, validate end-to-end, then remove dead local auth columns/tables in a cleanup migration at the end of the sprint.

## Open Questions

1. If Clerk remains the preferred provider, do we want the app to preserve backend-owned cookie auth, or are we intentionally accepting a token-in-SPA model?
2. If signup must still collect organization name, should local org creation remain the first-class app action, with provider provisioning happening after, instead of treating Clerk signup as the bootstrap event?

## Summary

The draft’s strategic conclusion, “buy rather than build,” is strong. The weak spots are in the migration mechanics: org bootstrap is underspecified, bearer-token auth is a larger contract break than the draft acknowledges, webhook-only sync is race-prone, the Clerk pricing basis needs correction, and the schema teardown is too aggressive for the first migration.
